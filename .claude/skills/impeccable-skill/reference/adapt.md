# Adapt：为不同上下文设计

> "适配是为新上下文重新思考体验，不是缩放像素。"

陷阱是将适配视为缩放。在适配之前理解三个维度：原始设计假设、目标平台约束和特定适配挑战。

---

## 评估适配挑战

**源上下文**：原始设计假设（屏幕大小、输入方法、连接速度、使用场景）。

**目标上下文**：设备类型、输入方法、屏幕约束、使用模式、用户期望。

**适配挑战**：什么在新环境中不适合、不能工作或感觉不合适？

---

## 平台特定策略

### 移动端适配（桌面 → 移动端）

**布局**：单列、垂直堆叠。没有侧边栏。内容优先。

**交互**：44×44px 最小触摸目标。没有悬停依赖的功能。主要操作的手指区域放置（屏幕底部，一只手可触及）。

**内容**：渐进披露——显示更少，按需揭示。压缩导航。

**导航**：Tab bar 或 hamburger；绝不暴露完整导航菜单。

### 平板适配（混合）

在纵向/横向方向之间适应的双列布局。支持触摸和指针输入。考虑主细节内容 的 split-view 模式。

### 桌面适配（移动端 → 桌面）

利用可用宽度的多列布局。悬停状态、键盘快捷键、右键上下文菜单。展开渐进披露——更多可以同时可见。

### 打印适配（屏幕 → 打印）

- 黑白布局；不带颜色测试
- 计划分页以避免内容中途分割
- 展开内容：显示完整 URL（不是超链接文本）
- 移除交互元素和导航
- 使用 `@media print` CSS

### 邮件适配（Web → 邮件）

- 最大宽度 600px
- 基于表的布局（CSS Grid 和 Flexbox 不是普遍支持）
- 内联样式（外部样式表经常被剥离）
- 单列以确保移动安全
- 在多个邮件客户端中测试（Outlook 是邮件的 IE）

## 实现适配

**断点**：内容驱动，不是设备驱动。从窄开始，拉伸直到设计破坏，在那里添加断点。三个标准断点（640、768、1024px）通常足够。

**输入方法检测**（比屏幕大小对交互模式更可靠）：
```css
@media (pointer: coarse) { /* 触摸 */ }
@media (hover: none) { /* 无悬停 */ }
```

**触摸目标**：小视觉元素仍然需要 44px 最小点击区域——使用 padding 或伪元素。

**缺口设备**：`env(safe-area-inset-*)` 配合 `viewport-fit=cover`。

---

## 验证适配

在真实设备上测试——不仅仅是 DevTools。模拟遗漏真实的触摸交互、真实 CPU/内存约束和网络延迟。测试：
- 多个屏幕大小和方向
- 触摸 vs 指针输入
- 多个浏览器
- 不同操作系统（文本渲染不同）
- 慢网络条件
- 每个平台仅用键盘导航
