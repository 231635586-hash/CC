# Persona-Based Design Testing（基于角色的设计测试）

通过五个不同的用户原型评估界面，每个都暴露不同的失败模式。

---

## 五个角色

### Alex — 不耐烦的高级用户
优先考虑效率和快捷方式。因不必要的步骤或缓慢动画而放弃产品。

**红旗**：强制 onboarding、无键盘快捷键、冗余确认对话框、缓慢过渡。

### Jordan — 困惑的初次使用者
需要明确的指导和清晰的标签。因假设先前知识的界面而放弃。

**红旗**：行话、未标记的图标、无空状态指导、不清楚的下一步。

### Sam — 依赖无障碍的用户
需要键盘导航、适当的 ARIA 标签和 WCAG AA 颜色对比度（最低 4.5:1）。

**红旗**：缺少焦点指示器、仅颜色信息、非语义 HTML、破碎的 Tab 顺序。

### Riley — 深思熟虑的压力测试者
系统地探索边界情况和意外输入，以识别静默失败和状态管理问题。

**红旗**：无错误状态、静默失败、返回导航后状态破碎、刷新时数据丢失。

### Casey — 分心的移动用户
单手操作手机，需要手指区域操作，并需要跨中断的状态保持。

**红旗**：小触摸目标、重要操作在顶部角落、无状态保持、悬停依赖交互。

---

## 按界面类型的角色选择

| 界面类型 | 推荐角色 |
|----------------|---------------------|
| 结账/支付流程 | Casey、Riley、Jordan |
| 仪表盘/数据表 | Alex、Sam |
| Onboarding/注册 | Jordan、Casey |
| 设置/配置 | Alex、Riley |
| 公共营销页面 | Jordan、Sam |
| 复杂表单/向导 | Riley、Jordan、Sam |

---

## 自定义角色

当存在 PRODUCT.md 或设计上下文部分时，从项目特定受众数据中派生 1–2 个额外角色。基于实际用户特征的测试优于推测。

---

## 如何使用

以每个相关角色浏览界面。对于每个步骤，问：这个角色会在这里成功吗？什么会让他们沮丧或阻止他们？记录具体摩擦点——不是一般印象——并为 critique 报告标记角色名称。
